    async function getDefaultFetcher(): Promise<BundlerFetcher> {

    **Replacement:**
    ```typescript
    async function getDefaultFetcher(): Promise<BundlerFetcher> {
      // Route through ProviderRouter (Phase 16) to gain Shyft fallback and
      // throw-on-exhaustion semantics. The router exposes getTransactionDetails(sig),
      // which structurally satisfies BundlerFetcher.getTransaction(sig) — both
      // take a string and return a Promise<{ signature, nativeTransfers? }>.
      // The `as unknown as BundlerFetcher` cast bridges the method-name mismatch
      // (getTransactionDetails vs getTransaction) without requiring a wrapper object.
      const { sharedProviderRouter } = await import('../fetchers/providers/index.js');
      return {
        getTransaction: (signature: string) =>
          sharedProviderRouter.getTransactionDetails(signature),
      };
    }
    ```

    **Rationale for explicit wrapper over `as unknown as BundlerFetcher`:**
    The PATTERNS.md suggests `return sharedProviderRouter as unknown as BundlerFetcher`. That cast works structurally but exposes the entire ProviderRouter surface (including `fetchSwapHistory`, `getStatus`, etc.) to anything that downcasts. The explicit `{ getTransaction: (sig) => router.getTransactionDetails(sig) }` adapter is:
    1. More type-safe (no `unknown` cast needed — TypeScript checks the method signature directly)
    2. Clearer at the call site (the method-name bridge is explicit code, not an invisible cast)
    3. Identical performance (no per-call wrapper allocation — the arrow function is created once per `getDefaultFetcher()` invocation, which itself is called at most once per detector run)
