<feature>
  <name>Sniper Detector (DETC-03)</name>
  <files>src/detection/sniper.ts, src/detection/__tests__/sniper.test.ts</files>
  <behavior>
    Input: DetectorConfig { walletAddress, thresholdMultiplier }
    Returns: DetectorResult

    Algorithm:
    1. Query swaps table for the target wallet's buy transactions with their slot values
    2. For each token_mint the wallet has bought:
       a. Find the minimum slot across ALL wallets' buys for that token_mint (the approximate launch slot)
       b. Count how many OTHER distinct wallets bought that token (require >= SNIPER.MIN_OTHER_WALLETS_FOR_BASELINE)
          If fewer than MIN_OTHER_WALLETS_FOR_BASELINE other wallets bought it → skip this token (no reliable baseline)
       c. Compute wallet_entry_offset = wallet's slot - launch_slot
       d. If wallet_entry_offset <= SNIPER.FIRST_BLOCK_WINDOW_SLOTS → count as a first-block entry
    3. Collect: total_tokens_with_baseline = tokens where we had enough other wallets to establish baseline
    4. Collect: first_block_entries = count of first-block entries from eligible tokens
    5. Apply thresholds with multiplier:
       - effective_suspected = SNIPER.MIN_LAUNCHES_SUSPECTED * thresholdMultiplier
       - effective_tokens_suspected = SNIPER.MIN_TOKENS_FOR_SUSPECTED (not multiplied — total token count is objective)
       - effective_review = SNIPER.MIN_LAUNCHES_REVIEW * thresholdMultiplier
       - effective_confirmed_count = SNIPER.MIN_LAUNCHES_CONFIRMED * thresholdMultiplier
       - effective_confirmed_rate = SNIPER.MIN_RATE_CONFIRMED (rate not multiplied)
       - effective_tokens_for_rate = SNIPER.MIN_TOKENS_FOR_RATE_CONFIRMED
    6. Determine confidence:
       - flagged=false if first_block_entries < effective_suspected OR total_tokens_with_baseline < effective_tokens_suspected
       - 'suspected' if first_block_entries >= effective_suspected AND total_tokens_with_baseline >= effective_tokens_suspected
       - 'review' if first_block_entries >= effective_review AND total_tokens_with_baseline >= effective_tokens_for_review
       - 'confirmed_suspicious' if first_block_entries >= effective_confirmed_count
           OR (first_block_entries / total_tokens_with_baseline >= effective_confirmed_rate AND total_tokens_with_baseline >= effective_tokens_for_rate)
    7. Return:
       - evidenceSummary: { first_block_entries, total_tokens, rate: first_block_entries/total_tokens }
       - evidenceDetail: { eligible_tokens: [{ token_mint, launch_slot, wallet_entry_slot, offset }] }

    SQL pattern for step 2:
    ```sql
    SELECT
      s.token_mint,
      MIN(s.slot) as launch_slot,
      MIN(CASE WHEN s.wallet_address = :addr THEN s.slot END) as wallet_entry_slot,
      COUNT(DISTINCT CASE WHEN s.wallet_address != :addr THEN s.wallet_address END) as other_wallet_count
    FROM swaps s
    WHERE s.side = 'buy'
    GROUP BY s.token_mint
    HAVING wallet_entry_slot IS NOT NULL
    ```

    Test cases (use in-memory db or mock):
    - wallet with 0 eligible tokens → flagged=false
    - wallet with 5 first-block entries across 8 tokens → suspected
    - wallet with 4 first-block entries across 8 tokens (below MIN_LAUNCHES=5) → flagged=false
    - wallet with 8 first-block entries across 10 tokens → review
    - wallet with 12 first-block entries across 15 tokens → confirmed_suspicious (by count)
    - wallet with 13 first-block entries across 15 tokens (87% rate, above 80%) → confirmed_suspicious (by rate)
    - wallet with 5 first-block entries but all tokens have <3 other wallets → flagged=false (no baseline)
    - wallet with 5 first-block entries, thresholdMultiplier=2.0 → flagged=false (needs 10 for suspected now)
    - wallet that bought 15 tokens ALL at launch but thresholdMultiplier=2.0 → review (needs 16 for confirmed_count, but rate still 100% >= 80%)
  </behavior>
  <implementation>

<feature>
  <name>Wash Trader Detector (DETC-04)</name>
  <files>src/detection/wash-trader.ts, src/detection/__tests__/wash-trader.test.ts</files>
  <behavior>
    Input: DetectorConfig { walletAddress, thresholdMultiplier }
    Returns: DetectorResult

    Algorithm:
    1. Get the target wallet's buy transactions from swaps (wallet_address = target, side = 'buy')
    2. For each buy (up to WASH_TRADER.MAX_HELIUS_FETCHES_PER_WALLET total fetches across all transactions):
       a. Fetch the Helius transaction for the buy's tx_signature to get tokenTransfers and nativeTransfers
       b. Find any wallets in tokenTransfers[].toUserAccount (token recipients from target wallet's buy tx)
       c. Check if any such recipient wallet (wallet B) sold the same token_mint — query swaps WHERE wallet_address = B AND token_mint = target_token AND side = 'sell' AND timestamp > buy_timestamp AND timestamp < buy_timestamp + RELATIONSHIP_WINDOW_DAYS*86400*1000
       d. If wallet B sold: fetch B's sell transaction from Helius, check if nativeTransfers show SOL going back to the target wallet (or nativeTransfers show a common sender flowing to both)
       e. "Circular pattern confirmed" = target buys token → token transferred to B → B sells token within window
          Note: requiring explicit SOL-back transfer is ideal but may not always appear in nativeTransfers; counting the buy→transfer→sell chain alone (without SOL return) is acceptable as circumstantial evidence.
    3. Count independent circular patterns (each must involve a different token_mint OR a different wallet B)
    4. Apply thresholds with multiplier:
       - effective_suspected = WASH_TRADER.MIN_CIRCULAR_PATTERNS_SUSPECTED * thresholdMultiplier
       - effective_review = WASH_TRADER.MIN_CIRCULAR_PATTERNS_REVIEW * thresholdMultiplier
       - effective_confirmed = WASH_TRADER.MIN_CIRCULAR_PATTERNS_CONFIRMED * thresholdMultiplier
    5. Return DetectorResult with appropriate confidence
       - evidenceSummary: { circular_patterns: N, related_wallets: [...] }
       - evidenceDetail: { patterns: [{ token_mint, wallet_b, buy_tx, sell_tx }] }

    Relationship definition (Claude's Discretion implemented here):
    A wallet B is "related" to wallet A if ANY of these are true within RELATIONSHIP_WINDOW_DAYS:
    - B appears in nativeTransfers of A's transactions (SOL flows between them)
    - B appears in tokenTransfers of A's transactions (token flows between them)
    This definition is used only to identify candidate wallets B to check; the circular pattern itself is the primary evidence.

    Test cases (mock Helius fetcher, in-memory db):
    - wallet with 0 circular patterns → flagged=false
    - wallet with 1 pattern (below MIN_CIRCULAR_PATTERNS_SUSPECTED=2) → flagged=false
    - wallet with 2 independent patterns → suspected
    - wallet with 4 patterns → review
    - wallet with 7 patterns → confirmed_suspicious
    - wallet with 2 patterns but thresholdMultiplier=2.0 → flagged=false (needs 4 for suspected)
    - wallet with patterns but all same token_mint (not independent) → count as 1 pattern, not N
    - wallet with Helius calls exceeding MAX_HELIUS_FETCHES_PER_WALLET → caps and returns with partial evidence (adds note in evidenceSummary: { fetch_limit_hit: true })
  </behavior>
  <implementation>
    After RED, implement detectWashTrader:
    - Accept optional { db, fetcher } deps for testability
    - Use createHeliusFetcher() through the existing heliusQueue singleton to respect rate limits
    - Use WASH_TRADER constants from thresholds.ts
    - Enforce MAX_HELIUS_FETCHES_PER_WALLET cap — most important guard given rate limit concerns
    - Return partial evidence with fetch_limit_hit=true when capped (do not crash or throw)
    - Independence check: patterns on the same token_mint with the same wallet_b count as 1 pattern
  </implementation>
</feature>
