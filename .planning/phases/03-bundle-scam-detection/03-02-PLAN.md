<feature>
  <name>Bundler Detector (DETC-01)</name>
  <files>src/detection/bundler.ts, src/detection/__tests__/bundler.test.ts</files>
  <behavior>
    Input: DetectorConfig { walletAddress, thresholdMultiplier }
    Returns: DetectorResult

    Algorithm:
    1. Query swaps table for all (slot, token_mint) pairs where:
       - side = 'buy'
       - The target wallet is one of the buyers
       - COUNT(DISTINCT wallet_address) >= BUNDLER.MIN_WALLETS_IN_SAME_SLOT in that slot+token_mint group
       These are "coordination candidate events"

    2. For each coordination candidate event involving the target wallet:
       - Fetch the Helius transaction for the target wallet's swap tx_signature (using createHeliusFetcher().getTransaction or fetchTransactions)
       - Inspect nativeTransfers in the response — find the common sender (fromUserAccount) that funded multiple participants
       - Exclude any sender in BUNDLER.KNOWN_SYSTEM_ACCOUNTS
       - If a common funder is found, count this as one confirmed "coordination event with shared funder"
       - Cap total Helius fetches at BUNDLER.MAX_HELIUS_FETCHES

    3. Apply threshold with multiplier:
       - effective_suspected = BUNDLER.MIN_EVENTS_SUSPECTED * thresholdMultiplier
       - effective_review = BUNDLER.MIN_EVENTS_REVIEW * thresholdMultiplier
       - effective_confirmed = BUNDLER.MIN_EVENTS_CONFIRMED * thresholdMultiplier

    4. Return DetectorResult:
       - flagged=false if coordination events < effective_suspected
       - confidence='suspected' if events >= effective_suspected
       - confidence='review' if events >= effective_review
       - confidence='confirmed_suspicious' if events >= effective_confirmed
       - evidenceSummary: { coordination_events: N, shared_funder_count: M, sample_tokens: [...] }
       - evidenceDetail: { events: [{ slot, token_mint, co_buyers: [...], shared_funder }] }

    Test cases (using db mocks / in-memory db):
    - wallet with 0 coordination events → flagged=false
    - wallet with 1 event (below MIN_EVENTS_SUSPECTED=2) → flagged=false
    - wallet with 2 events, shared funder, thresholdMultiplier=1.0 → suspected
    - wallet with 3 events, shared funder → review
    - wallet with 5 events, shared funder → confirmed_suspicious
    - wallet with 4 events but thresholdMultiplier=2.0 (after clear) → suspected (threshold raised: needs 4 for review)
    - wallet with 3 events but shared funder is JUP6... (excluded) → flagged=false
    - wallet with 2 events but fewer than MIN_WALLETS_IN_SAME_SLOT=3 co-buyers → flagged=false

    Implementation note: For unit tests, mock the db and Helius fetcher. Test files should use jest.mock() or pass a db/fetcher parameter via dependency injection. Design detectBundler to accept an optional { db, fetcher } parameter for testability. In production use, default to the singleton db and fetcher from their respective modules.
  </behavior>

</feature>

<feature>
  <name>Dev Wallet Detector (DETC-02)</name>
  <files>src/detection/dev-wallet.ts, src/detection/__tests__/dev-wallet.test.ts</files>
  <behavior>
    Input: DetectorConfig { walletAddress, thresholdMultiplier }
    Returns: DetectorResult

    Algorithm:
    1. Get all distinct token_mints from swaps for the wallet (side='buy')
    2. For each token_mint not already checked (check via wallet_flags WHERE detector='dev_wallet' for this wallet):
       - Fetch the earliest transactions for that token_mint via Helius
         (use createHeliusFetcher().fetchTransactions(tokenMint, limit=20, oldest-first) to find creation tx)
       - Find the transaction where the mint was created: look for the feePayer or the account that appears in tokenTransfers as a sender of this token to multiple parties — this is the deployer
       - Check tokenTransfers in that transaction and the next DEV_WALLET.DEPLOYER_TRANSFER_LOOKFORWARD_TXS transactions:
         if any tokenTransfers[].toUserAccount === walletAddress AND tokenTransfers[].mint === tokenMint, that is the deployer transfer
       - Also check if tokenTransfers[].fromUserAccount matches the feePayer of the creation transaction (the deployer)
    3. If one confirmed deployer → walletAddress transfer found:
       - Return immediately with confidence = DEV_WALLET.CONFIDENCE_ON_FIRST_SIGNAL ('confirmed_suspicious')
       - evidenceSummary: { token_mint, deployer_address, transfer_tx_signature }
       - evidenceDetail: { full tokenTransfer object }
    4. thresholdMultiplier does not affect dev wallet (one signal is always sufficient — confirmed by research)
    5. If no deployer transfer found across all tokens → flagged=false

    Limitation acknowledged: Helius `GET /v0/addresses/{mint}/transactions` may or may not work for mint addresses vs wallet addresses. Implementation-time: try fetching from the mint address. If it returns empty, fall back to checking the first swap transaction for the wallet on that token_mint (look at nativeTransfers and tokenTransfers on the swap tx itself — sometimes the deployer transfer is in the same transaction).

    Test cases:
    - wallet with token received directly from deployer in creation tx → confirmed_suspicious immediately
    - wallet with token received in tx 2 after creation (within LOOKFORWARD_TXS=3) → confirmed_suspicious
    - wallet that bought token on DEX (not from deployer) → flagged=false
    - wallet with no buys → flagged=false
    - wallet with deployer transfer but thresholdMultiplier=4.0 → STILL confirmed_suspicious (threshold not applicable here)

    Design for testability: accept optional { db, fetcher } parameter like bundler detector.
  </behavior>
  <implementation>
    After RED, implement detectDevWallet:
    - Use createHeliusFetcher() defaulting to singleton
    - For each token mint, fetch transactions — extract feePayer and tokenTransfers
    - Compare toUserAccount in tokenTransfers to walletAddress
    - Return on FIRST match — no need to check all tokens after first flag
    - Use DEV_WALLET constants from thresholds.ts
    - Document the Helius mint-address fetch limitation in a comment
  </implementation>
</feature>
